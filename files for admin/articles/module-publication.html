Magento Marketplace is a Magento module store where the merchant can find the right solution for building a business, just like the developer can find the tools to expand the standard Magento feature.


Perspective Studio team has the experience of developing and publishing a module on the Magento Marketplace platform and wants to introduce you all the stages of this process.

by Evgen Mozok

Module Development

Within this case, Perspective Studio team worked with an unusual client. As a rule, we are module developers for companies with online shops at Magento. In this case, the client's business builts on the referral system basis, which means they're looking for partners who help in the sales of goods by placing links to the sites.

At the same time, the company cooperates with various ecommerce systems, including Magento. We have developed a referral system module for Magento 1 and Magento 2. The module tracks the buyer's behaviour: the transition to the site and further actions on the order registration. The purpose of such a module is to collect data about buyers who visited this site by the referral link.

Terms of development

Development lasted two months. Initially estimated the task at 60-80 hours. The duration of development was influenced by the peculiarities of the technical task and changes to it within the work process. There is well-known research by Steve McConel, that the later the client makes changes to the initial technical task during development, the more money and time he needs for implementation. Therefore, we recommend this legislation be presented to the client before the start of work.

Development stages

The publication of the module at the Magento Marketplace provides two types of reviews: technical and marketing. Development is carried out in accordance with the standards and requirements of Magento. Development consists of several stages:

  1. Preparation of technical materials

  2. Preparation of marketing materials

  3. Creating and filling an account

  4. Module publication

  5. Passing technical and marketing review.

We will tell you about each stage in detail (for convenience all the screens to the text are hidden).

  1. Preparation of technical materials

    The publication on the Marketplace implies the passing a Technical review. Magento Coding standards and Technical Review Guidelines should be followed during the development of the module. These guidelines are mandatory for extensions published on the Marketplace since code style will be automatically checked through Code Sniffer. We recommend configuring PHP_CodeSniffer and using Magento Coding Standard.

    Composer.json

    The module must contain a valid composer.json file with all required dependencies. The composer.json file specifies:

    • name
    • type
    • version

    The composer.json does not declare:

    • extra.map
    • extra.magento-root-dir
    • restrict versions of PHP that do not meet the Magento restrictions
    • does not declare dependence on:
      • magento/magento-composer-instроaller
      • magento/magento2-base
      • magento/product-community-edition
      • magento/magento2-ee-base
      • magento/product-enterprise-edition
    • does not use * as a dependency version
    • require inline alias

    We also recommend you validate composer.json by installing your extension on your local project. Simply install an additional repository with path to the repository directory of the Repositories - Composer module.

    Unit Tests

    If you decide to publish your extension to the Magento Marketplace, it is time to start writing the unit tests. Despite the fact that there are no direct requirements for the number of tests, it is recommended to cover with unit tests from 60% to 80% of your code. You can usually write tests after the main code, but this is a good time to try.

    Additional

    During module development, it is crucial to find vulnerable places. Malware Scan checks the extension, all media files and documents for malicious code or links. All code and marketing content submitted to the Magento Marketplace is also checked to be unique and not duplicated code from an already known extension. The module must be installed without errors on the specified versions of Magento and run without errors in production mode with the Varnish Cache enabled. Autotests check if store pages get into the cache and come back from the Varnish Cache.

  2. Preparation of marketing materials

    To publish on the Magento Marketplace, you need to present your module professionally. DevDocs describes how to prepare a marketing review. You need to make the following marketing materials:

    • vendor logo 255px by 255px
    • module logo 240px by 240px (must be different from vendor logo)
    • a short and detailed description of the product as part of the marketing processes module images (Minimum of two, maximum of 15.5 MB in JPG, JPEG or PNG format)
    • video link (optional)
    • browser compatibility (optional)
    • pricing policy support rules (optional)
    • additional materials.

  3. Creating and filling in an account

    To publish a module, you need to have a Magento/Adobe commerce account. If you already have it, use it. But it needs to be further configured. Register for Adobe Commerce Marketplace - Developer Portal. You need to fill in Account, Marketplace, and Tax Form Information for the module to be published.

    Our explanations about Marketplace Information:

    • Screen Name - generated automatically
    • Vendor Name - must match the vendor name in composer.json of your modules
    • Vendor logo - 255px by 255px, at loading checked on "harmful" materials
    • Partner status - whether you are a business partner of Magento
    • Privacy Policy URL - a link to the privacy policy. We used the link on the client's current website.

    When you complete Tax Form Information, you must specify whether you pay taxes in the United States and provide additional documents.

  4. Extension publication

    By clicking Adobe Commerce Marketplace - Developer Portal you can view existing extensions and add a new one. I will write in detail about our experience of publication:

    • we review and agree with the process
    • create a new extension:
    • title (header decoders: extension content | adobe commerce developer guide)
    • how to sell - one-time purchase or subscription
    • if extension includes external integration, additional information must be provided
    • you can sell existing versions and add a new version on the extensions page
    • adding a new version:
      • the version number must match the version in composer.json
      • the time when the extension will be launched: after a successful validation or a specific date
    • the extension version page contains:
      • the first block - general information, name, generated id
      • the second block is the status of technical and marketing review
      • the third block is technical and marketing information about the module

  5. Passing technical and marketing review.

    Technical and marketing reviews can be passed independently of each other. If all the reviews are successful, you will see a lot of green icons on the screen.

    However, there are might be mistakes that lead to failure.

    Be sure to prepare the User Guide as it is required for Manual QA.

    Pay enough attention to the long Description form of your module. Explain what the module can do for a merchant and how it works. List key features. Describe what changes can be made in the interface or UX of the site, or can be made to the settings. You can give examples of cases when merchant can use the module. Leave a clear understanding of how its website will look and function if it buys your extension.

    Also, note that to change the date of module publication, it is necessary to remove it from the marketing review, change the type/date of publication and re-submit to the marketing review.

Module publication: What's next?

The module was published on the Magento Marketplace. Both the merchant and the developer can download and apply for extensions in the online shop. However, publication at the Magento Marketplace does not mean that the work is completed. What's next?

During the operation of the module, additional nuances may arise related to the peculiarities of a particular store and environment.

We also recommend checking the installation and initial setup process. In this case, after checking on our test environment, the customer has passed the process of installation of the module in its environment.

Modifications are possible with the release of a new version of Magento. The module must be checked for compatibility with the new version. Despite the fact that we used best practices and technical guidelines.

The referral system module must support not only new versions but also older versions of Magento. After all, there are online shops that are not updated. The more complex the function of the module, the more problems with the previous versions and updates will arise.
